Project Deep Dive: Weekly Arisan System

Background

This idea came from my mother's weekly arisan group. Every week, one person took turns recording payments in a notebook. The problems: frequent miscalculations, missed entries, and end-of-period reconciliation meant starting from scratch. I realized this could be automated. The app lets admins mark PAID/UNPAID, view each participant's payment history, and export Excel reports to share via WhatsApp.

Technical Decisions
  • Next.js for frontend + API routes — one language (TypeScript) end to end.
  • Supabase for PostgreSQL database + Auth + real-time — minimal infrastructure overhead.
  • Unique constraint (participant_id, week_number) to prevent duplicate payments.
  • Excel export via ExcelJS for offline recap needs.
Challenges

The biggest challenge was maintaining payment data integrity — ensuring no duplicate or missed entries. Solved with a unique constraint at the database level. Another challenge was keeping the UI simple, since the primary users are not tech-savvy. Privacy was also a concern: payment amounts and winners should not be publicly accessible.

Tech Stack: Next.js 16, Supabase (PostgreSQL + Auth), TypeScript, Tailwind CSS v4, ExcelJS

Project Deep Dive: Resident Data System

Background

This system was originally requested by the local neighborhood head. Resident data was recorded in a single ledger kept at their home. Anyone needing data — for official letters or surveys — had to visit in person, and the ledger was often incomplete or missing pages. I saw this could be solved with a simple system: resident data stored in a database, searchable, filterable, and exportable anytime. Top priorities: data security and limited admin access.

Technical Decisions & Security
  • Next.js + Supabase with Row Level Security for granular data access.
  • Supabase Auth for admin authentication — only registered users can manage data.
  • Public timeline shows only basic info; detailed data stays in the admin dashboard.
  • Excel export for neighborhood report needs.
  • Block filter + pagination for efficient data navigation.
Privacy Considerations

Resident data is sensitive. This system follows the least privilege principle — each user can only access data necessary for their role. The public repo is a mirror with no real data.

Tech Stack: Next.js 16, Supabase (PostgreSQL + Auth + RLS), TypeScript, Tailwind CSS v4, ExcelJS, Vitest
